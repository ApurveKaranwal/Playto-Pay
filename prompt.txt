# CORE REQUIREMENTS

## 1. Ledger System (SOURCE OF TRUTH)

* Use a LedgerEntry table
* Types: CREDIT, DEBIT, HOLD, RELEASE
* DO NOT store balance as a column
* Balance must ALWAYS be computed using DB aggregation

Invariant:
SUM(CREDIT + RELEASE) - SUM(DEBIT + HOLD) = BALANCE

---

## 2. Money Rules

* Use integer paise ONLY
* Use BigIntegerField
* NEVER use float
* NEVER use Decimal unless absolutely necessary

---

## 3. Concurrency (CRITICAL)

Scenario:

* Balance = 10000 paise
* Two payouts of 6000 paise arrive simultaneously

Expected:

* Only ONE succeeds
* Other fails cleanly

MANDATORY IMPLEMENTATION:

* Use transaction.atomic()
* Use select_for_update()
* Balance check MUST be inside transaction
* NO check-then-update outside transaction

---

## 4. Idempotency (CRITICAL)

Each request has:
Idempotency-Key (header)

Rules:

* Same key → SAME response
* No duplicate payouts
* Use unique constraint on (merchant, idempotency_key)

Edge case:

* Two identical requests arrive at same time
* Must not create duplicate records

---

## 5. Payout State Machine

Allowed:

* pending → processing → completed
* pending → processing → failed

Forbidden:

* completed → anything
* failed → completed
* any backward transition

You MUST enforce this in code.

---

## 6. Background Worker (Celery ONLY)

Flow:

1. Pick pending payouts
2. Move → processing
3. Simulate:

   * 70% success
   * 20% fail
   * 10% hang

On success:

* Add DEBIT ledger entry

On failure:

* Add RELEASE ledger entry

ALL updates must be inside transaction.atomic()

---

## 7. Retry Logic

* If stuck in processing > 30 seconds
* Retry with exponential backoff
* Max retries = 3
* Then mark failed + RELEASE funds

---

## 8. APIs

POST /api/v1/payouts
Headers:

* Idempotency-Key

Body:

* merchant_id
* amount_paise
* bank_account_id

GET /balance
GET /payouts

---

## 9. Tests (MANDATORY)

1. Concurrency test:

* Two simultaneous payout requests
* Only one succeeds

2. Idempotency test:

* Same request twice
* Same response
* Only one DB entry
